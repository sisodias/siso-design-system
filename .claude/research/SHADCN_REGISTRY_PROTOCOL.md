# shadcn Registry Protocol — Deep Research

**Date:** 2026-04-23
**CLI version covered:** shadcn CLI v4 (March 2026) + v3 (August 2025) history
**Sources:** all primary shadcn docs + live JSON schema at `https://ui.shadcn.com/schema/registry-item.json`

---

## Bottom Line Up Front

**Does our `registry-item.json` format work with `npx shadcn@latest add` as of April 2026?**

**YES, with two caveats.** The per-component `registry-item.json` file (served at e.g. `https://registry.siso.dev/r/button.json`) is the correct primitive — the CLI fetches a single item JSON, validates it against the JSON Schema, and installs from it. Caveat 1: `tailwind.config` is **deprecated for Tailwind v4** — use `cssVars.theme` instead. Caveat 2: for `@siso/component-name` shorthand, users must add a `registries` block to their `components.json` — `@siso` cannot be registered centrally without submitting to the open-source registry index.

## Q1: HTTP mechanics

| Requirement | Value |
|---|---|
| Method | `GET` |
| User-Agent sent by CLI | `shadcn` |
| Accept header sent | `application/vnd.shadcn.v1+json, application/json;q=0.9` |
| Expected Content-Type back | `application/json` |
| CORS | Not required for CLI (Node process). Required for browser tooling. |
| HTTPS | Strongly recommended |

## Response shape — required vs optional

Schema: `https://ui.shadcn.com/schema/registry-item.json` (JSON Schema draft-07)

**Minimum required:** only `name` + `type`.

| Field | Type | Required | Notes |
|---|---|---|---|
| `$schema` | string | No | Advisory only |
| `name` | string | **YES** | Unique in registry |
| `type` | enum | **YES** | 12 valid values |
| `title`, `description`, `author` | string | No | |
| `dependencies` | string[] | No | `name@version` supported |
| `devDependencies` | string[] | No | |
| `registryDependencies` | string[] | No | Bare slugs, `@ns/name`, or URLs |
| `files` | object[] | No | The actual payload |
| `tailwind` | object | No | **DEPRECATED v4** — use `cssVars.theme` |
| `cssVars` | object | No | `{theme, light, dark}` |
| `css` | object | No | At-rules, keyframes, utilities |
| `envVars` | object | No | → `.env.local`, never overwrites |
| `meta` | object | No | Fully open KV — our secret weapon |
| `docs`, `categories` | — | No | |
| `font` | object | Conditional | `type === "registry:font"` only |

**Type enum:** `registry:base` | `block` | `component` | `font` | `lib` | `hook` | `ui` | `page` | `file` | `style` | `theme` | `item`

## Q2: `registry.json` (collection) vs `registry-item.json` (single)

| File | Purpose | Who uses it |
|---|---|---|
| `registry.json` | Root index listing all items, generated by `shadcn build` | `search`/`list` commands + namespace routing |
| `registry-item.json` | Installable payload for one item | `npx shadcn add <url>` fetches this |

**Our approach of per-component files is correct.** We additionally need a root `registry.json` for `search`/`list`.

## Q3: Kit / Bundle install

No dedicated `kit.json` format. Three mechanisms:
1. `registryDependencies` chain — one item lists many, recursive install
2. Multiple URL args — `npx shadcn add url1 url2 url3`
3. `registry:base` type (CLI v4, March 2026) — packs entire design system into one payload. **This is our kit primitive.**

---

## Q4: Style tokens + Tailwind config merging

On install:
1. Reads target `components.json` for CSS path
2. Merges `cssVars.theme` into `@theme {}` (Tailwind v4)
3. Merges `cssVars.light`/`dark` into `:root {}`/`.dark {}`
4. Appends `css` rules to target CSS
5. Legacy: extends `tailwind.config.js` (v3 only)
6. Writes `envVars` to `.env.local` — never overwrites

## Q5: Namespace prefixes — can we register `@siso`?

Source: https://ui.shadcn.com/docs/registry/namespace

**Decentralized.** Any namespace by anyone. Collisions prevented locally.

**User config:**
```json
{
  "registries": {
    "@siso": "https://siso-design-system.vercel.app/r/{name}.json"
  }
}
```

**With auth:**
```json
{
  "registries": {
    "@siso": {
      "url": "https://...r/{name}.json",
      "headers": { "Authorization": "Bearer ${SISO_TOKEN}" }
    }
  }
}
```

URL placeholders: `{name}`, `{style}`.

**Global registration:** ui.shadcn.com/docs/registry/registry-index — optional community directory.

## Q6: Auth

All via `components.json`, not CLI flags:

| Method | Config |
|---|---|
| Bearer | `headers.Authorization: "Bearer ${TOKEN}"` |
| API key | `headers.X-API-Key: "${KEY}"` |
| Basic | `headers.Authorization: "Basic ${b64}"` |
| Query param | `params.token: "${TOKEN}"` |

Env vars expanded at runtime. Never logged. Error codes: 401, 403, 404, 429.

---

## Q7: Version pinning

**No URL path versioning.** Uses query params:
```json
{
  "params": { "version": "${SISO_VERSION:-stable}" }
}
```

CLI v4 added `--diff` flag. `npx shadcn add` always re-fetches — no client cache.

## Q8: Discovery UX

| Command | Function |
|---|---|
| `npx shadcn search` | All registries |
| `npx shadcn search @ns` | Namespace list |
| `npx shadcn search @ns --query "button"` | Filtered |
| `npx shadcn list` | Alias |
| `npx shadcn view <url>` | Preview without install |

`search` reads root `registry.json` collection. registry.directory is the community surface.

## Q9: Error handling

| Scenario | Behavior |
|---|---|
| Missing npm dep | Auto-installs via detected pm |
| File exists | Prompts to overwrite; `--overwrite` skips |
| Missing `registryDep` slug | 404 with slug name |
| Circular dep | Detected, error |
| Missing auth env var | Shows exact `${VAR_NAME}` |
| Missing shadcn primitive | Auto-installs recursively |

---

## Q10: Caching

**No client-side cache.** Every `add` re-fetches. Server-side `Cache-Control`/`ETag` your responsibility.

## Q12: Our competitive advantage

With 2,200 classified + `ai_summary` + curation tags + ratings + thumbnails:

| Our capability | shadcn hook | Advantage |
|---|---|---|
| `ai_summary` per component | `description` + `docs` | LLM-generated, richer than hand-written |
| Multi-taxonomy categories | `categories[]` | Cross-dim tagging |
| Quality ratings | `meta.siso_rating` | Agents prioritize high-quality |
| Thumbnails | `meta.siso_thumbnail` | Displayed in v0/shadcn browser |
| Source tracking | `author` + `meta.siso_source` | Attribution + update tracking |
| Multi-style variants | `{style}` URL placeholder | Auto-serve right variant |
| `@siso` namespace | `registries` in `components.json` | One-line setup |

**The `meta` field is our secret weapon.** Fully open `additionalProperties: true`:

```json
{
  "meta": {
    "siso_rating": 4.2,
    "siso_category": ["data-table", "dashboard"],
    "siso_thumbnail": "https://...thumbs/qualified-table.png",
    "siso_ai_summary": "...",
    "siso_source": "21st.dev",
    "siso_installs": 847
  }
}
```

Agents using shadcn MCP server receive this metadata → our registry is the most signal-rich option.
